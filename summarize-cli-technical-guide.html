Overview & Architecture

The summarize CLI is a sophisticated Node.js-based tool that orchestrates multiple content extraction pipelines with various LLM providers. It features a modular architecture with a separate daemon server for browser extension integration, two-tier caching, and support for diverse content types including websites, YouTube videos, podcasts, and local files.

🖥️

CLI Interface

Command-line tool built with Commander.js for argument parsing and user interaction.

🔄

Content Pipelines

Specialized extraction flows for websites, YouTube, documents, and media files.

🤖

LLM Gateway

Unified interface supporting OpenAI, Anthropic, Google, xAI, and OpenRouter providers.

💾

Two-Tier Caching

SQLite database for structured data, file-based cache for media downloads.

🌐

Daemon Server

HTTP server bridging browser extension to CLI capabilities via SSE streaming.

⚙️

Configuration

JSON5-based settings with model rules, API keys, and provider preferences.

High-Level System Architecture

Key architectural principle: The system uses a gateway-style model ID format (`/`) that enables transparent routing between different LLM providers with automatic fallback mechanisms and unified streaming interfaces.

CLI Command Flow

The CLI follows a clear execution path from user invocation to final output. Understanding this flow is essential for debugging and extending the tool's functionality.

Entry Points

The CLI has three primary entry points that handle initialization, orchestration, and execution:

📍

src/cli.ts

Entry Point

Top-level entry point that imports and calls runCliMain() with process.argv and environment details.

🔧

src/cli-main.ts

Initialization

Handles error piping, loads .env files, and calls runCli() for core logic execution.

⚙️

src/run/runner.ts

Orchestration

Contains runCli() which parses arguments, resolves input, and routes to appropriate flows.

Execution Pipeline

The command execution follows a sequential pipeline with pre-flight checks, input resolution, and flow routing:

TypeScript · CLI Flow Pseudocode// User invokes: summarize <url|file>

async function runCli(argv) {
  // Step 1: Pre-flight checks
  if (isHelpRequest(argv)) return showHelp();
  await refreshFreeModels();  // Update available models
  await handleDaemonCommands(argv);  // Install/status/run

  // Step 2: Parse arguments with Commander
  const program = buildProgram();
  const options = program.parseAsync(argv);

  // Step 3: Resolve input type
  const input = await resolveRunInput(options);
  // Returns: { type: 'url' | 'file' | 'stdin', value, ... }

  // Step 4: Route to appropriate flow
  if (input.type === 'url') {
    return await runUrlFlow(input, options);
  } else if (input.type === 'file' || input.type === 'stdin') {
    return await handleFileInput(input, options);
  }
}

Website Content Extraction

Website summarization relies on Mozilla Readability, a library originally created for Firefox's Reader View. It extracts the main article content by analyzing the HTML DOM and identifying which elements contain actual content versus navigation, ads, and sidebars.

Website Extraction Pipeline

Extraction Process

The website extraction follows a multi-stage process with automatic fallback mechanisms:

Primary Extraction (Readability)

Mozilla Readability parses the HTML DOM and extracts article content by removing navigation, ads, sidebars, and comments. It returns clean title and body text.

Fallback Mechanism (Firecrawl)

If Readability extraction produces less than MIN_READABILITY_CONTENT_CHARACTERS, the system automatically falls back to Firecrawl API for more robust extraction.

TypeScript · Website Content Extractionimport { JSDOM } from 'jsdom';
import { Readability } from '@mozilla/readability';

async function extractWebsiteContent(url: string) {
  // Step 1: Fetch HTML
  const response = await fetch(url);
  const html = await response.text();

  // Step 2: Parse with jsdom (virtual DOM)
  const dom = new JSDOM(html);
  const document = dom.window.document;

  // Step 3: Extract with Readability
  const reader = new Readability(document);
  const article = reader.parse();

  // Step 4: Check content quality
  if (!article || article.textContent.length < MIN_CHARS) {
    // Fallback to Firecrawl
    return await extractWithFirecrawl(url);
  }

  // Step 5: Convert to Markdown (optional)
  return {
    title: article.title,
    content: article.textContent,
    // markdown: convertHtmlToMarkdown(article.content)
  };
}

Markdown mode options: The --markdown-mode flag controls HTML conversion with options like readability (default), auto, llm, or off. Each mode uses different strategies for converting HTML to Markdown format.

YouTube Transcript Pipeline

YouTube transcript acquisition uses multiple methods with automatic fallback, from free web scraping to paid APIs. Each method has different trade-offs in speed, reliability, and coverage.

ModeMethodSpeedCostCoverageReliability
youtubei YouTube internal API Fast Free Limited Medium
captionTracks Embedded caption files Fast Free Limited High
yt-dlp Download + transcribe Slow Free Universal High
apify Web scraping service Fast Paid High High

Mode Fallback Order

The --youtube auto mode (default) attempts methods in sequence until one succeeds:

YouTube Transcript Fallback Flow
TypeScript · YouTube Transcript Fetchingasync function fetchTranscript(
  videoId: string,
  mode: 'auto' | 'web' | 'yt-dlp' | 'apify'
): Promise<Transcript> {

  if (mode === 'auto' || mode === 'web') {
    // Try youtubei (internal API)
    const youtubeiResult = await fetchTranscriptFromTranscriptEndpoint(videoId);
    if (youtubeiResult) return youtubeiResult;

    // Try captionTracks (embedded captions)
    const captionResult = await fetchTranscriptFromCaptionTracks(videoId);
    if (captionResult) return captionResult;
  }

  if (mode === 'auto' && canRunYtDlp()) {
    // Download audio and transcribe with Whisper
    const ytdlpResult = await tryYtDlpTranscript(videoId);
    if (ytdlpResult) return ytdlpResult;
  }

  // Final fallback: Apify scraping service
  if (mode === 'auto' || mode === 'apify') {
    return await fetchTranscriptWithApify(videoId);
  }

  throw new Error('No transcript available');
}

Universal coverage advantage: The yt-dlp mode works on any YouTube video, even those without captions, by downloading audio and transcribing it with Whisper. This makes it the ultimate fallback at the cost of speed and processing requirements.

LLM Gateway & Provider Integration

The LLM gateway provides a unified interface for multiple AI providers through a "gateway-style" model ID format. It handles provider selection, streaming responses, and automatic fallback between providers.

Supported Providers

🟣

🔶

⚫

💠

Model Attempts System

The gateway uses a sophisticated "model attempts" system that transforms user configuration into an ordered list of validated attempts:

TypeScript · Model Attempt Pipelineinterface ModelAttempt {
  transport: 'native' | 'openrouter' | 'cli';
  userModelId: string;
  llmModelId: string;
  requiredEnv: string[];
}

async function buildModelAttempts(
  config: Config,
  inputType: InputType
): ModelAttempt[] {
  const attempts: ModelAttempt[] = [];

  // Step 1: Rule resolution - match input to config rules
  const rules = resolveRulesForInput(config.model, inputType);

  // Step 2: CLI prepend (if enabled)
  if (config.cli.enabled) {
    attempts.push(...buildCliAttempts(config.cli.enabled));
  }

  // Step 3: Build attempts from candidates
  for (const candidate of rules.candidates) {
    const attempt = buildAttemptForCandidate(candidate);

    // Step 4: Validate (API keys, token limits)
    if (validateAttempt(attempt, config.env)) {
      attempts.push(attempt);

      // Step 5: Add OpenRouter fallback for native providers
      if (attempt.transport === 'native') {
        const openRouterAttempt = buildOpenRouterFallback(attempt);
        if (openRouterAttempt) attempts.push(openRouterAttempt);
      }
    }
  }

  return attempts;
}

Automatic fallback behavior: The system attempts each model in sequence until one succeeds. If streaming times out or fails, it automatically falls back to non-streaming mode before trying the next provider in the list.

Caching System

The caching system uses a two-tier architecture with SQLite for structured data and a file-based cache for media downloads. Both caches support size limits, TTL expiration, and automatic eviction.

🗄️

SQLite Cache

Structured Data

Stores transcripts, extracted content, and summaries in a single SQLite database at ~/.summarize/cache.sqlite.

  • Website extraction results (Readability/Firecrawl)
  • YouTube transcripts (all modes)
  • LLM-generated summaries with metadata
  • Slide extraction data
📁

Media Cache

File Storage

Stores downloaded audio/video files from yt-dlp in ~/.summarize/cache/media/.

  • YouTube audio downloads for transcription
  • Podcast audio files
  • Video files for slide extraction
  • Temporary processing files

Cache Key Generation

All cache keys use SHA256 hashes with format versioning to ensure invalidation on code changes:

TypeScript · Cache Key Generationimport { createHash } from 'crypto';

const CACHE_FORMAT_VERSION = 'v1';

// Transcript cache key: URL + namespace + file modification time
function buildTranscriptCacheKey(
  url: string,
  namespace?: string,
  fileMtime?: number
): string {
  const data = JSON.stringify({ CACHE_FORMAT_VERSION, url, namespace, fileMtime });
  return createHash('sha256').update(data).digest('hex');
}

// Summary cache key: content hash + prompt hash + model + length + language
function buildSummaryCacheKey(
  contentHash: string,
  promptHash: string,
  model: string,
  lengthKey: string,
  languageKey: string
): string {
  const data = JSON.stringify({
    CACHE_FORMAT_VERSION,
    contentHash,
    promptHash,
    model,
    lengthKey,
    languageKey
  });
  return createHash('sha256').update(data).digest('hex');
}

Daemon Server Architecture

The daemon server bridges the browser extension to CLI capabilities, running as an HTTP server on 127.0.0.1:8787. It handles heavy processing tasks that can't be performed in the browser environment.

Purpose and Benefits

The daemon enables the Chrome extension to access local tools and APIs that browsers cannot directly reach:

🔧

Local Tool Access

Executes yt-dlp, ffmpeg, whisper.cpp, and tesseract for media processing and OCR.

🔑

API Key Security

Keeps API keys on the local machine rather than exposing them through browser extensions.

⚡

Heavy Processing

Offloads CPU-intensive tasks like transcription and content extraction from the browser.

🔄

SSE Streaming

Provides real-time progress updates via Server-Sent Events to the extension UI.

💾

Shared Cache

Extension uses the same SQLite cache as the CLI, avoiding redundant work.

🏃

Background Service

Runs as a system service (LaunchAgent, systemd, or Scheduled Task) for persistence.

TypeScript · Daemon Server Setupimport { createServer } from 'http';

const DAEMON_PORT = 8787;

async function runDaemonServer(config: DaemonConfig) {
  const server = createServer(async (req, res) => {
    const url = new URL(req.url, `http://localhost:${DAEMON_PORT}`);

    switch (url.pathname) {
      case '/health':
        handleHealth(req, res);
        break;

      case '/v1/summarize':
        if (!verifyAuthToken(req)) {
          res.statusCode = 401;
          res.end(JSON.stringify({ error: 'Unauthorized' }));
          return;
        }
        await handleSummarizeRequest(req, res);
        break;

      default:
        res.statusCode = 404;
        res.end('Not Found');
    }
  });

  server.listen(DAEMON_PORT, '127.0.0.1');
  console.log(`Daemon running on http://127.0.0.1:${DAEMON_PORT}`);
}

Authentication mechanism: The daemon uses a bearer token generated during installation, stored in ~/.summarize/config.json for the daemon and chrome.storage.local for the extension. This token is required for all /v1/ endpoints.

Best Practices

When building your own summarization tool based on these patterns, consider these proven practices from the summarize CLI implementation.

🎯

Use Gateway-Style Model IDs

Format models as <provider>/<model> to enable transparent routing between providers. This abstraction makes adding new providers trivial without changing user-facing code.

🔄

Implement Automatic Fallback

Never rely on a single extraction method or API. Always have fallback chains with clear error boundaries. The summarize CLI tries 4+ methods for YouTube transcripts alone.

💾

Cache Aggressively

Cache everything that's expensive: API calls, network requests, media downloads. Use SHA256-based keys with format versions to ensure cache invalidation works correctly.

📋

Separate Config from Code

Use a declarative configuration file (JSON5/YAML) for model selection rules, provider settings, and behavior. This lets users customize without code changes.

🌐

Support Multiple Content Types

Don't limit to just URLs. Support files, stdin, and specialized inputs like YouTube URLs. Auto-detect input type to reduce user friction.

🎨

Enable Streaming by Default

Users expect real-time feedback. Stream LLM responses and progress updates. Fall back to non-streaming only on errors or timeouts.

Extraction strategy: Start with free, fast methods (Mozilla Readability, web scraping) and fall back to paid APIs (Firecrawl, Apify) only when necessary. This keeps costs low while maintaining reliability.

Error handling approach: Implement attempt-level retry with exponential backoff for transient failures, then cross-attempt fallback before surfacing errors to the user. This handles most temporary issues automatically.
