What Are AI Agents?

An AI agent is an LLM-powered system that can autonomously perceive its environment, reason about a goal, plan a sequence of actions, execute those actions using tools, and iterate until the goal is achieved — all with minimal human intervention.

Unlike a simple chatbot (single-turn request → response), an agent operates in a loop: it thinks, acts, observes the result, and thinks again.

The Agent Loop
🧠

LLM as the Brain

The language model does reasoning, planning, and decision-making. It reads context and decides what action to take next, or if the goal has been reached.

🔧

Tools / Actions

Agents can call external tools — web search, databases, APIs, code execution, file I/O — to get information and cause effects in the real world.

💾

Memory

Agents maintain state across turns: short-term (context window), working memory, long-term (vector DB / database), and episodic/procedural stores.

🎯

Goal / Task

Agents are driven by a goal or objective. This can be explicit ("book a flight") or implicit (a system prompt defining an assistant's purpose).

🔄

Autonomy

Unlike one-shot prompts, agents decide independently how many steps to take and which tools to use, looping until the task is complete or impossible.

👁️

Environment

The environment is everything the agent can observe: chat history, tool outputs, databases, APIs, files, and other agents' messages.

Key mental model: Think of the LLM as a very smart CPU — it can only "run" one instruction at a time. The agent loop is the operating system scheduling work, storing results in memory, and calling I/O (tools) as needed.

Short-Term Memory: Managing the Context Window

# Short-term memory = the messages array sent to the API
# Strategies: sliding window, summarization, priority pruning

import anthropic
from collections import deque

client = anthropic.Anthropic()

class ShortTermMemory:
    def __init__(self, max_turns=20, summarize_at=15):
        self.history = deque(maxlen=max_turns)
        self.summarize_at = summarize_at
        self.summary = ""

    def add(self, role, content):
        self.history.append({"role": role, "content": content})
        if len(self.history) >= self.summarize_at:
            self._compress()

    def _compress(self):
        # Summarize old turns to free context space
        old = list(self.history)[:10]
        prompt = f"Summarize this conversation: {old}"
        r = client.messages.create(
            model="claude-opus-4-5", max_tokens=500,
            messages=[{"role": "user", "content": prompt}]
        )
        self.summary += r.content[0].text + "\n"
        for _ in range(10):
            self.history.popleft()

    def get_messages(self):
        msgs = list(self.history)
        if self.summary:
            # Prepend summary as context
            msgs.insert(0, {
                "role": "user",
                "content": f"[Prior context]: {self.summary}"
            })
        return msgs

Long-Term Memory: Vector Database (RAG)

Python · Long-Term Memory with Vector DBimport anthropic
from sentence_transformers import SentenceTransformer
import chromadb

client = anthropic.Anthropic()
embed_model = SentenceTransformer("all-MiniLM-L6-v2")
chroma = chromadb.Client()
collection = chroma.get_or_create_collection("agent_memory")

def save_to_longterm(content: str, metadata: dict):
    embedding = embed_model.encode(content).tolist()
    collection.add(
        documents=[content],
        embeddings=[embedding],
        metadatas=[metadata],
        ids=[f"mem_{hash(content)}"]
    )

def retrieve_relevant(query: str, n_results=5) -> list[str]:
    query_embed = embed_model.encode(query).tolist()
    results = collection.query(
        query_embeddings=[query_embed],
        n_results=n_results
    )
    return results["documents"][0]

def agent_with_memory(user_query: str, stm: ShortTermMemory):
    # Pull relevant memories into context
    memories = retrieve_relevant(user_query)
    memory_ctx = "\n".join(memories)
    
    stm.add("user", user_query)
    
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=1024,
        system=f"You are a helpful agent.\n\nLong-term memories:\n{memory_ctx}",
        messages=stm.get_messages()
    )
    
    answer = response.content[0].text
    stm.add("assistant", answer)
    save_to_longterm(user_query + " → " + answer, {"type": "qa"})
    return answer

Defining and Using Tools

Python · Tool Definitions + Agent Loopimport anthropic, json, requests

client = anthropic.Anthropic()

# Tool schemas — Claude will decide when/how to call these
TOOLS = [
    {
        "name": "web_search",
        "description": "Search the web for current information.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "Search query"}
            },
            "required": ["query"]
        }
    },
    {
        "name": "calculate",
        "description": "Evaluate a mathematical expression.",
        "input_schema": {
            "type": "object",
            "properties": {
                "expression": {"type": "string", "description": "Math expression"}
            },
            "required": ["expression"]
        }
    },
    {
        "name": "write_file",
        "description": "Write content to a file.",
        "input_schema": {
            "type": "object",
            "properties": {
                "filename": {"type": "string"},
                "content": {"type": "string"}
            },
            "required": ["filename", "content"]
        }
    }
]

# Tool executor — map LLM's tool call to real functions
def execute_tool(name: str, inputs: dict) -> str:
    if name == "web_search":
        # Real impl: call SerpAPI, Tavily, etc.
        return f"[Search results for '{inputs['query']}': ...]"
    elif name == "calculate":
        return str(eval(inputs["expression"]))  # use safer eval in prod
    elif name == "write_file":
        with open(inputs["filename"], "w") as f:
            f.write(inputs["content"])
        return f"File '{inputs['filename']}' written."
    return "Unknown tool"

# The core agentic loop
def run_agent(task: str, max_iterations: int = 10) -> str:
    messages = [{"role": "user", "content": task}]
    
    for i in range(max_iterations):
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=4096,
            tools=TOOLS,
            messages=messages
        )
        
        # Done — no more tool calls
        if response.stop_reason == "end_turn":
            return response.content[0].text
        
        # Process tool calls
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result
                })
        
        # Feed results back into context
        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user", "content": tool_results})
    
    return "Max iterations reached"

# Run it!
result = run_agent("Research the current gold price, calculate how much 15oz would cost, and save the result to gold_report.txt")
print(result)

Cognitive Architectures

A cognitive architecture defines how an agent thinks and structures its reasoning. Different architectures trade off between simplicity, quality, and computational cost.

ReAct

Reasoning + Acting interleaved. The agent writes a "thought" before each action, making reasoning transparent and controllable. Most common pattern.

Plan-and-Execute

Planner creates a full task list upfront; Executor runs each step. Good for complex, predictable workflows. Faster per step.

Tree of Thought (ToT)

Explores multiple reasoning branches in parallel. Best for problems requiring search through many possible solutions (math, planning).

Reflexion

Agent reflects on failures and self-improves across episodes. Stores verbal feedback to avoid repeating mistakes. Great for coding tasks.

LATS (MCTS)

LLM-powered Monte Carlo Tree Search. Extremely powerful for agentic search problems. Highest quality, highest cost.

Critic / Verifier

A second LLM call critiques the first's output before acting. Reduces hallucinations and mistakes. Simple but effective for high-stakes tasks.

ReAct Pattern (most widely used)

Python · ReAct Architecturedef react_agent(task: str):
    """
    ReAct loop: Thought → Action → Observation → repeat
    The system prompt shapes the LLM to output structured thoughts.
    """
    system = """You are a ReAct agent. For each step, output:
Thought: [your reasoning about what to do next]
Action: [tool_name]
Action Input: [JSON input for the tool]

When done, output:
Thought: I now have enough information.
Final Answer: [your answer]"""
    
    messages = [{"role": "user", "content": f"Task: {task}"}]
    
    while True:
        resp = client.messages.create(
            model="claude-opus-4-5", max_tokens=1024,
            system=system, messages=messages,
            stop_sequences=["Observation:"]  # stop before observation
        )
        output = resp.content[0].text
        
        if "Final Answer:" in output:
            return output.split("Final Answer:")[1].strip()
        
        # Parse Action + Input
        action = parse_action(output)
        observation = execute_tool(action["tool"], action["input"])
        
        # Append thought/action + observation to history
        messages.append({"role": "assistant", "content": output})
        messages.append({"role": "user", "content": f"Observation: {observation}"})

Plan-and-Execute

Python · Plan-and-Execute Architecturedef plan_and_execute(task: str):
    # Step 1: Planner generates a task list (one LLM call)
    plan_resp = client.messages.create(
        model="claude-opus-4-5", max_tokens=1024,
        system="Create a numbered step-by-step plan to accomplish the task. Output JSON array.",
        messages=[{"role": "user", "content": task}]
    )
    steps = json.loads(plan_resp.content[0].text)
    # e.g. ["Search for X", "Summarize results", "Write report"]
    
    context = {"task": task, "results": []}
    
    # Step 2: Execute each step in sequence
    for i, step in enumerate(steps):
        exec_resp = client.messages.create(
            model="claude-opus-4-5", max_tokens=2048,
            system=f"You are executing step {i+1}/{len(steps)}: {step}\nPrevious results: {context['results']}",
            tools=TOOLS,
            messages=[{"role": "user", "content": f"Execute: {step}"}]
        )
        # ... handle tool calls, collect result ...
        context["results"].append({"step": step, "result": exec_resp.content[0].text})
    
    return context

Choosing an architecture: Use ReAct for general-purpose agents. Use Plan-and-Execute when tasks are highly structured. Use Reflexion for iterative improvement (coding, writing). Use ToT/LATS for reasoning-heavy problems where cost is acceptable.

LangGraph: Stateful Graph-Based Agent

Python · LangGraph stateful agentfrom langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

class AgentState(TypedDict):
    messages: Annotated[list, operator.add]
    tool_calls: list
    final_answer: str

def call_llm(state: AgentState):
    response = client.messages.create(
        model="claude-opus-4-5", max_tokens=1024,
        tools=TOOLS, messages=state["messages"]
    )
    return {"messages": [response], "tool_calls": response.content}

def run_tools(state: AgentState):
    results = []
    for block in state["tool_calls"]:
        if hasattr(block, "name"):
            result = execute_tool(block.name, block.input)
            results.append({"tool_use_id": block.id, "content": result})
    return {"messages": [{"role": "user", "content": results}]}

def should_continue(state: AgentState):
    last = state["messages"][-1]
    if last.stop_reason == "end_turn":
        return "end"
    return "tools"

graph = StateGraph(AgentState)
graph.add_node("llm", call_llm)
graph.add_node("tools", run_tools)
graph.set_entry_point("llm")
graph.add_conditional_edges("llm", should_continue, {"tools": "tools", "end": END})
graph.add_edge("tools", "llm")
agent = graph.compile()

When to use raw API: For production systems, raw API gives you precise control over costs, latency, and behavior. Use frameworks to prototype quickly; migrate to direct API calls for performance-critical paths.

Multi-Agent Architectures

Single agents hit limits: context window size, specialization, parallelism, and reliability. Multi-agent systems solve this by distributing work across specialized agents that communicate.

Multi-Agent Topology Patterns

Orchestrator-Worker Pattern (most common production pattern)

Python · Orchestrator-Worker Multi-Agentclass OrchestratorAgent:
    def run(self, task: str) -> str:
        # Orchestrator decides which workers to call
        plan = self._plan_subtasks(task)
        results = {}
        
        for subtask in plan:
            worker = WORKER_REGISTRY[subtask["agent"]]
            results[subtask["id"]] = worker.execute(
                task=subtask["task"],
                context={k: results[k] for k in subtask.get("depends_on", [])}
            )
        
        return self._synthesize(task, results)

    def _plan_subtasks(self, task: str) -> list:
        resp = client.messages.create(
            model="claude-opus-4-5", max_tokens=1024,
            system="""You are an orchestrator. Decompose tasks into subtasks.
Available agents: researcher, coder, writer, critic, data_analyst.
Return JSON: [{"id":"t1","agent":"researcher","task":"...","depends_on":[]}]""",
            messages=[{"role": "user", "content": task}]
        )
        return json.loads(resp.content[0].text)

Agent Communication: A2A Protocol

Python · Agent-to-Agent communication via shared message busimport asyncio
from dataclasses import dataclass

@dataclass
class AgentMessage:
    sender: str
    receiver: str     # or "broadcast"
    content: str
    message_type: str # "task" | "result" | "query" | "error"

class MessageBus:
    def __init__(self):
        self.queues: dict[str, asyncio.Queue] = {}
    
    def register(self, agent_id: str):
        self.queues[agent_id] = asyncio.Queue()
    
    async def send(self, msg: AgentMessage):
        if msg.receiver == "broadcast":
            for q in self.queues.values():
                await q.put(msg)
        else:
            await self.queues[msg.receiver].put(msg)
    
    async def receive(self, agent_id: str) -> AgentMessage:
        return await self.queues[agent_id].get()

bus = MessageBus()
# Agents communicate asynchronously via the bus

Production Architecture with Queuing + State

Python · Production-ready agent servicefrom fastapi import FastAPI, BackgroundTasks
from redis import Redis
import uuid

app = FastAPI()
redis = Redis(host="redis", decode_responses=True)

@app.post("/tasks")
async def submit_task(task: str, bg: BackgroundTasks):
    task_id = str(uuid.uuid4())
    redis.hset(f"task:{task_id}", mapping={
        "status": "queued", "task": task
    })
    bg.add_task(process_task, task_id, task)
    return {"task_id": task_id}

async def process_task(task_id: str, task: str):
    redis.hset(f"task:{task_id}", "status", "running")
    try:
        result = await run_agent_async(task)
        redis.hset(f"task:{task_id}", mapping={
            "status": "completed", "result": result
        })
    except Exception as e:
        redis.hset(f"task:{task_id}", mapping={
            "status": "failed", "error": str(e)
        })

@app.get("/tasks/{task_id}")
async def get_task(task_id: str):
    return redis.hgetall(f"task:{task_id}")

Recommended stack for most teams: FastAPI + Redis queues + Cloud Run (or ECS) for stateless agents. Add Temporal for durability if tasks exceed 5+ minutes or must survive failures. Use Langfuse or LangSmith for observability.

Production Best Practices

Human-in-the-Loop

For high-stakes actions (send email, execute trade, delete data), require explicit human approval. Build interrupt points into the agent loop.

Observability

Log every LLM call, tool execution, and decision. Use Langfuse, LangSmith, or OpenTelemetry to trace agent runs end-to-end.

Guardrails

Validate tool inputs/outputs. Limit tool permissions (least privilege). Set iteration limits. Add input/output classifiers for harmful content.

Cost Control

Cache tool results and LLM responses. Use cheaper models for simple sub-tasks. Set hard token limits. Monitor cost-per-task.

Failure Handling

Agents fail gracefully: implement retry with backoff, fallback strategies, and clear error messages that the LLM can reason about.

Evals

Build an evaluation harness with golden datasets. Test agent behavior on edge cases. Monitor production performance over time.
